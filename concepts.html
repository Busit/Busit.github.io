Learn the basic concepts

What is a connector ?
A connector is a little piece of code that can do stuff. See them as independant boxes that users can link to each other in order to create scenarios (aka: flows).
For instance:
  1. Collect news from CNN
  2. Then look for interesting keywords
  3. Then translate it in another language
  4. And finally send it to my email




As you can guess, there are 3 distinct types of connectors:
  1. Data producers usually bring data from your devices or services into Busit. (i.e.: CNN)
  2. Data transformers usually modify or enhance data it receives and passes it on if necessary. (i.e.: translate)
  3. Data consumers usually send data it receives to your devices or services. (i.e.: email)
Note that some connectors can be of several types at once, for instance, the twitter connector allows to gather tweets (and thus act as a data producer because it brings those into Busit) but also to post new tweets (and thus act as a data consumer because it sends information out of Busit). This depends on the interfaces defined for that connector.

What is an instance ?
An instance of a connector is a user's copy of that connector whereas the connector itself is more like a template. Instances are usually configured and linked in a flow so they are ready to be used. Users can have multiple instances of the same connector and of course, instances of different connectors as well.

We speak about connectors when it is not bound to a user, and about instances when it is.

What is a flow ?
A flow is the actual scenario that a user creates when linking instances of different connectors. A flow usually starts with a producer in order to get data, then any number of transformers, then a consumer.

Users can create as many flows as they want by reusing existing instances or new ones.

For instance, get my Facebook messages and save them on Dropbox:

Get all messages from Facebook 

Save to Dropbox 

Rarely, there can also be partial flows which do not start with a producer, in that case, data has to come in the flow by other means: using the API to push data directly on an instance.

What is an interface ?
Every connector have one or more interfaces that can emit or receive data. See them as different doors that allows to enter or exit the connector.
  1. Producers have output interfaces because you get data from them, data comes out of the connector.
  2. Consumers have input interfaces because you send data to it, data gets inside the connector.
  3. Transformers have both input and output interfaces because you put data in them and they return it after transformation, data gets in and then out of the connector.

Meanwhile, since connectors can be multiple things at once (consumer and producer) they may have input and output interfaces and thus look like a transformer... In that case, simply refer to the logical behavior of the connector: the twitter connector has an input interface to post tweets and also an independant output interface to get new tweets, though, it does not transform tweets; whereas the translation connector has both an input and an output interface so the text that goes in is directly translated and comes out, this is a transformer.

There can be multiple input or output interfaces on a single connector, so in order to make it clear each have a name and a description.

For instance, the Netatmo connector has 3 output interfaces:

Netatmo Weather Station 
Temperature
The outside air temperature in Celcius
CO2
The CO2 level in ppm
Noise
The noise level in dB

Note that there are some special interfaces that the user can customize or that can be triggered automatically.

What is a custom interface ?
A custom interface can be any input or output that require user configuration to work properly. Hence, you can have multiple time the same interface with different configuration.
For instance, the email connector has a single custom input interface that requires an email address as configuration.
Therefore, users can add several email addresses if needed:

bob@example.com

What is an automatic interface ?
An automatic (or cron) interface will be activated at regular interval. It is typically designed for producers that pull data from your devices or services. Moreover, it can have multiple applications for time related matters: birthdays, alarm clock, reminders,...

What about push and pull ?
We talk about push and pull when it comes to exchange data accross devices or services. We always look at the origin of the data:
  • When the device that has the data sends it directly to Busit, we say it pushes its data to the platform. On the other way around, when Busit has data and sends it directly to an external service, Busit will push data to that service.
    My phone pushes the data to Busit.
    Here is the data !
    
  • When the device that has the data keeps it and waits for Busit to collect it every once in a while (see automatic interfaces) then we say that Busit pulls the data from that device.
    Busit pulls the data from my phone.
    Do you have some data ?
    
    Yes, here it is.
    

As you can guess, the ideal situation is always to push data because it is straight forward and allows realtime processing. However, this heavily depends on what the external devices or services support. On its side, Busit can collect data from external sources either in push or pull. However, Busit can only push data to external services. This is enforced by Busit's terms of use stating that we do not store data ; hence, we cannot keep it until some external service pulls it.

How to configure an instance ?
Some connectors require configuration parameters to work properly. For instance, the Facebook connector needs your security token to be able to post or read messages.
There are 2 distinct ways to configure an instance:
  1. If the configuration is basic, a simple form is automatically generated such that the user can set the value manually.
  2. If the configuration is more complex, then the developer of the connector can use an external configuration website that will guide the user through the process.
